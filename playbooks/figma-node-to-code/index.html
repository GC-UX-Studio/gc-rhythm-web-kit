Agent Prompts

Figma Node To Code Workflow

A structured prompt sequence for UX and engineering teams using Codex with Figma MCP. The operating model is always inspect -> classify -> validate -> implement, with token inspection used only when a mapping looks unclear.

Recommended Workflow

The default path should be:

  1. 1Component Map
  2. 2Component Validate
  3. 3Component Code
  4. 4Module Map
  5. 5Module Validate
  6. 6Module Code
  7. 7Layout Map
  8. 8Layout Validate
  9. 9Layout Code

Optional Diagnostic

Run Token Audit only when spacing, typography, color, radius, or stroke looks ambiguous.

Token Audit
Diagnostic Only
0. Token Audit

Use this only when spacing, typography, color, radius, or stroke looks unclear.

Prompt
Inspect this Figma node and identify only token-level mappings.

Output only:
- spacing tokens
- typography tokens (role first, primitive only if needed)
- color tokens (component token, then semantic token, then primitive only if required)
- radius and stroke tokens if applicable
- ambiguities

Do not infer components.
Do not suggest variants.
Do not write code.
Components
Classify, Validate, Implement
1. Component Map

Map the node to the nearest valid Rhythm Web component before any code is written.

Prompt
Inspect this Figma node and map it to the nearest valid Rhythm Web component.

Decision rules:
- classify first
- prefer existing documented component variants
- do not invent a new variant
- if the node does not cleanly map, report the nearest valid pattern and the mismatch

Output only:
- component name
- variant
- size
- visible state
- supported state
- tokens used
- Figma details to ignore
- mismatches or ambiguity
- confidence: high, medium, or low

Do not write code.
  • Visible state is what Figma shows.
  • Supported state is what the system can actually implement.
2. Component Validate

Confirm the mapping is system-backed before it becomes implementation input.

Prompt
Validate this component mapping against Rhythm Web rules.

Check:
- component exists
- variant exists
- size exists
- state is supported
- tokens are system-backed
- ignored Figma details are correctly excluded

Output only:
- approved or rejected
- corrections if needed
- remaining mismatch notes

Do not write code.
3. Component Code

Generate code only after the component mapping is approved.

Prompt
Using the approved component mapping, generate code for this component.

Requirements:
- implement using Rhythm Web patterns and tokens
- do not invent new variants, props, or styling
- preserve supported interaction intent
- ignore unsupported Figma details
- match the target environment exactly: [repo-native code / HTML-CSS-JS / React / etc.]

Output:
- short mapping summary
- final code
- mismatch note if applicable
Modules
Compose From Approved Components
4. Module Map

Define composition and layout without reinterpreting child components.

Prompt
Inspect this parent Figma node and compose it using existing Rhythm components.

Decision rules:
- reuse previously approved component mappings
- do not reinterpret components already defined
- identify only composition and layout structure
- if a child does not map cleanly, flag it instead of inventing

Output only:
- component composition list
- layout structure (stack, row, grid, split, etc.)
- spacing and layout tokens
- responsive considerations
- mismatches or ambiguity
- confidence: high, medium, or low

Do not write code.
5. Module Validate

Verify that module composition stays inside the Rhythm system boundary.

Prompt
Validate this module composition against Rhythm Web rules.

Check:
- child components are reused consistently
- spacing uses documented tokens
- layout structure is system-consistent
- responsive behavior does not redefine internal component behavior
- no one-off composition patterns were introduced without need

Output only:
- approved or rejected
- corrections if needed
- remaining mismatch notes

Do not write code.
6. Module Code

Generate module code from the approved composition, not from raw Figma layout styling.

Prompt
Generate code for this module using the approved component composition.

Requirements:
- compose from existing components
- use spacing and layout tokens only
- avoid one-off layout values
- keep structure portable
- do not redefine internal component styling unless required by the target environment

Output:
- structure summary
- final code
- mismatch note
Layout
Place Approved Modules
7. Layout Map

Define page or section structure only after modules are stable.

Prompt
Inspect this layout and define page or section structure only.

Decision rules:
- use approved modules as building blocks
- do not redefine internal component or module behavior
- focus only on module placement, spacing rhythm, and responsive structure

Output only:
- modules used
- layout pattern
- spacing system between modules
- responsive considerations
- mismatches or ambiguity
- confidence: high, medium, or low

Do not generate full code.
8. Layout Validate

Check layout consistency before generating section or page code.

Prompt
Validate this layout structure against Rhythm Web rules.

Check:
- module reuse is consistent
- spacing between modules uses documented tokens
- responsive structure is system-backed
- no page-specific hacks or one-off layout concepts were introduced

Output only:
- approved or rejected
- corrections if needed
- remaining mismatch notes

Do not write code.
9. Layout Code

Generate layout code from the approved module structure only.

Prompt
Generate layout code using the approved modules and layout structure.

Requirements:
- do not redefine internal components or modules
- focus on layout, spacing, and responsive structure only
- use Rhythm spacing and responsive tokens
- avoid page-specific hacks
- keep the code portable for the target environment

Output:
- layout summary
- final code
- mismatch note
